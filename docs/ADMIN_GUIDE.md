## 1. Роли и права доступа

В системе 4 роли. При регистрации пользователь получает роль **participant**. Остальные назначаются администратором вручную.

### Сводная таблица прав

| Возможность | Participant | Expert | Moderator | Admin |
|---|:---:|:---:|:---:|:---:|
| Публичные страницы сайта | ✓ | ✓ | ✓ | ✓ |
| Личный кабинет | ✓ | ✓ | ✓ | ✓ |
| Подача и управление своей заявкой | ✓ | ✓ | ✓ | ✓ |
| Просмотр всех заявок | — | ✓ | ✓ | ✓ |
| Смена статуса заявки | — | ✓ | ✓ | ✓ |
| Экспорт заявок в Excel | — | — | ✓ | ✓ |
| Дашборд и аналитика | — | — | ✓ | ✓ |
| Просмотр и редактирование пользователей | — | — | ✓ | ✓ |
| Просмотр обращений с сайта | — | — | ✓ | ✓ |
| Создание и редактирование новостей | — | — | ✓ | ✓ |
| Удаление новостей | — | — | — | ✓ |
| Управление документами | — | — | — | ✓ |
| Управление победителями | — | — | — | ✓ |
| Управление номинациями | — | — | — | ✓ |
| Управление конкурсами | — | — | — | ✓ |
| Смена роли пользователя | — | — | — | ✓ |
| Создание аккаунта эксперта | — | — | — | ✓ |
| Удаление пользователей | — | — | — | ✓ |
| Блокировка пользователей | — | — | ✓ | ✓ |

> **Источник:** `backend/src/*/**.controller.ts` — декораторы `@Roles()`

---

### Participant (участник)
Базовая роль, назначается автоматически при регистрации.

- Видит публичные страницы: главная, новости, документы, победители, эксперты
- Имеет личный кабинет `/cabinet`
- Может создавать, редактировать (только черновик) и подавать заявку
- Может загружать файлы только к своей заявке (максимум **6 файлов**, до **50 МБ** каждый)
- Панель администратора недоступна

---

### Expert (эксперт)
Назначается администратором для оценки заявок.

- Всё, что может участник
- Доступ к `/admin/applications` — просмотр всех заявок
- Может менять статус заявки (принять, отклонить и т.д.)
- При переходе на `/admin` автоматически перенаправляется на `/admin/applications`
- Аналитика, CMS, пользователи — **недоступны**

---

### Moderator (модератор)
Доверенный сотрудник с расширенными правами.

- Всё, что может эксперт
- Дашборд `/admin` с общей статистикой
- Аналитика `/admin/analytics`
- Просмотр и базовое редактирование профилей пользователей
- Блокировка пользователей (флаг `isActive`)
- Создание и редактирование новостей (удалить — не может)
- Экспорт заявок в Excel
- Просмотр и обработка обращений с сайта

**Модератор НЕ может:** управлять документами, победителями, номинациями, конкурсами, менять роли пользователей, создавать экспертов.

---

### Admin (администратор)
Полный доступ ко всем функциям системы.

- Всё, что может модератор
- Удаление новостей
- Полное управление документами (загрузка, редактирование, удаление)
- Полное управление победителями
- Полное управление номинациями
- Создание и активация конкурсов
- Смена роли любого пользователя
- Создание аккаунтов экспертов
- Управление профилем эксперта (фото, должность, биография, видимость)
- Удаление пользователей (кроме себя)

## 4. Управление заявками

**URL:** `/admin/applications`

### Фильтрация списка
- По номинации
- По статусу
- По университету
- Поиск по названию проекта / ФИО участника
- По конкурсу

### Детальная карточка заявки
Открывается по клику «Просмотр». Содержит:
- Название и описание проекта
- Ключевые слова, состав команды, научный руководитель
- Данные участника (ФИО, email, телефон, вуз, факультет)
- Прикреплённые файлы (скачать можно напрямую)
- Историю смены статусов с комментариями и временными метками
- Блок смены статуса

### Смена статуса
1. Открыть заявку
2. В блоке **«Смена статуса»** выбрать новый статус
3. При отклонении — обязательно заполнить **«Комментарий»** (участник получит его в письме)
4. Нажать **«Сохранить»**

Email-уведомление участнику отправляется автоматически, если SMTP настроен.

### Экспорт в Excel
Доступен модератору и администратору.
Кнопка «Экспорт в Excel» на странице списка заявок. Экспортирует все заявки с учётом текущих фильтров (до 10 000 строк).

Колонки: №, название проекта, ФИО, email, университет, номинация, статус, дата подачи.

## 5. Жизненный цикл заявки

```
Участник создаёт заявку
        ↓
    [ЧЕРНОВИК]  ←──────────────────┐
        ↓  участник подаёт         │ участник отзывает
  [НА ПРОВЕРКЕ] ───────────────────┘
        │
        ├──→ [ПРИНЯТА]    — прошла первичный отбор
        ├──→ [ОТКЛОНЕНА]  — не прошла (обязательно укажите причину)
        └──→ [ДОПУЩЕНА]   — допущена к очному этапу
                 │
                 ├──→ [ПОБЕДИТЕЛЬ]
                 └──→ [ПРИЗЁР]
```

| Статус | Код | Кто переводит | Участник может редактировать |
|---|---|---|:---:|
| Черновик | `draft` | Создаётся участником | ✓ |
| На проверке | `submitted` | Участник (подача) | — |
| Принята | `accepted` | Expert, Moderator, Admin | — |
| Отклонена | `rejected` | Expert, Moderator, Admin | — |
| Допущена к очному | `admitted` | Moderator, Admin | — |
| Победитель | `winner` | Moderator, Admin | — |
| Призёр | `runner_up` | Moderator, Admin | — |

**Важно:**
- Участник может отозвать заявку из статуса «На проверке» обратно в «Черновик»
- Moderator и Admin могут принудительно перевести заявку в любой статус
- История всех изменений статусов сохраняется и видна в карточке заявки

## 6. CMS — управление контентом

### 6.1 Новости — `/admin/cms/news`
**Доступ:** Moderator, Admin (удаление — только Admin)

| Поле | Описание |
|---|---|
| Заголовок | Отображается в списке и на странице новости |
| Slug | URL-адрес (`/news/slug`), автогенерируется из заголовка |
| Краткое описание | Анонс в карточке новости |
| Текст | Полный текст статьи |
| Обложка | Изображение (JPEG, PNG, WebP, GIF, HEIC — до 5 МБ). HEIC конвертируется в JPEG автоматически |
| Опубликовано | Галочка делает новость видимой на сайте |
| Конкурс | Привязка к конкурсу (необязательно) |

---

### 6.2 Документы — `/admin/cms/documents`
**Доступ:** только Admin

Раздел для загрузки документов участникам (положения, шаблоны, инструкции).

| Поле | Описание |
|---|---|
| Название | Отображаемое название файла |
| Файл | PDF, DOC/DOCX, XLS/XLSX, PPT/PPTX, ZIP, TXT, PNG, JPG — до **50 МБ** |
| Категория | Классификация документа |
| Опубликовано | Видимость на странице `/documents` |

Скачивание документов — публичный эндпоинт, авторизация не нужна.

---

### 6.3 Победители — `/admin/cms/winners`
**Доступ:** только Admin

Управляет галереей победителей на странице `/winners`.

| Поле | Описание |
|---|---|
| Название проекта | — |
| Команда | Имена участников |
| Описание | Краткое описание проекта |
| Год | Год проведения конкурса |
| Место | 1, 2 или 3 (отображается медалями 🥇🥈🥉) |
| Номинация | Привязка к номинации |
| Университет | Вуз команды |
| Фото | Изображение (JPEG, PNG, WebP, GIF, HEIC — до 5 МБ) |
| Конкурс | Привязка к конкурсу |

---

### 6.4 Номинации — `/admin/cms/nominations`
**Доступ:** только Admin

Номинации — категории, по которым подаются заявки.

| Поле | Описание |
|---|---|
| Название | Полное название номинации |
| Краткое название | Для отображения в компактных блоках |
| Описание | Условия номинации |
| Активна | Деактивированные номинации не отображаются в форме заявки |
| Порядок | Число для сортировки в списке |

---

### 6.5 Конкурсы — `/admin/cms/contests`
**Доступ:** только Admin

Конкурс — это сезон проведения. К конкурсу привязываются заявки, победители и новости.

| Поле | Описание |
|---|---|
| Название | Название сезона конкурса |
| Описание | Краткое описание |
| Дата начала | — |
| Дата окончания | — |
| Активен | Только один конкурс может быть активным одновременно |

**Активация конкурса:**
- Нажать «Сделать активным»
- Предыдущий активный конкурс деактивируется автоматически
- Все новые заявки участников привязываются к активному конкурсу

> Если активного конкурса нет — участники не смогут подать заявку.

## 7. Управление пользователями

### Просмотр пользователей — `/admin/users`
**Доступ:** Moderator, Admin

- Поиск по имени, email
- Фильтр по роли
- Просмотр полного профиля
- Редактирование базовых данных: ФИО, телефон, университет, факультет, город

### Блокировка пользователя
**Доступ:** Moderator, Admin
- В профиле пользователя снять галочку «Аккаунт активен»
- Заблокированный пользователь не может войти в систему

### Смена роли
**Доступ:** только Admin
- В профиле пользователя выбрать новую роль из списка

### Удаление пользователя
**Доступ:** только Admin
- Нельзя удалить себя
- Удаление необратимо

### Создание эксперта — `/admin/experts`
**Доступ:** только Admin

**Способ 1 — создать вручную:**
1. Открыть `/admin/experts`
2. Нажать «Добавить эксперта»
3. Заполнить: email, временный пароль, ФИО, должность, биографию
4. Передать эксперту логин и пароль

**Способ 2 — повысить существующего участника:**
1. Найти пользователя в `/admin/users`
2. Изменить роль на `expert`
3. Заполнить профиль эксперта (должность, биографию, фото) в `/admin/experts`

### Профиль эксперта
**Доступ:** только Admin

- **Должность** — отображается на странице `/experts`
- **Биография** — краткое описание
- **Фото** — JPEG, PNG, WebP, GIF, HEIC — до 5 МБ (HEIC конвертируется в JPEG)
- **Показывать на сайте** — управляет видимостью в публичном списке экспертов

## 10. Личный кабинет участника

**URL:** `/cabinet`

Для понимания пути участника при ответе на вопросы.

### Что видит участник
- Личные данные (ФИО, вуз, факультет, город, курс) — редактируемые
- Email — только для чтения, изменить нельзя
- Список своих заявок с текущими статусами и цветовыми метками

### Работа с заявкой

| Действие | Условие |
|---|---|
| Создать заявку | Должен существовать активный конкурс |
| Редактировать | Только статус `draft` |
| Загрузить файл | Только статус `draft`, максимум **6 файлов** |
| Удалить файл | Только статус `draft` |
| Подать заявку | Только статус `draft` |
| Отозвать заявку | Только статус `submitted` → возвращается в `draft` |
| Удалить заявку | Только статус `draft` |

### Категории файлов заявки
- `project_file` — файл проекта
- `document_scan` — скан документа
- `archive` — архив

## 11. Типичные задачи

### Принять или отклонить заявку
1. `/admin/applications` → найти заявку
2. Нажать «Просмотр»
3. Выбрать новый статус в блоке «Смена статуса»
4. При отклонении — **обязательно заполнить комментарий**
5. «Сохранить» — участник получит email (если SMTP настроен)

---

### Запустить новый сезон конкурса
1. `/admin/cms/contests` → «Создать конкурс» (название, даты)
2. «Сделать активным»
3. Проверить номинации: `/admin/cms/nominations` → активировать нужные
4. Опубликовать новость о старте: `/admin/cms/news`

---

### Добавить победителей
1. `/admin/cms/winners` → «Добавить победителя» для каждого призёра
2. Указать место, год, проект, команду, номинацию, фото
3. Победитель сразу появится на `/winners`
4. Дополнительно: сменить статус заявок победителей на `winner` / `runner_up`

---

### Зарегистрировать нового эксперта
1. `/admin/experts` → «Добавить эксперта»
2. Заполнить ФИО, email, пароль, должность
3. Передать эксперту логин и пароль
4. При необходимости заполнить биографию и загрузить фото

---

### Заблокировать пользователя
1. `/admin/users` → найти пользователя
2. В профиле снять галочку «Аккаунт активен»
3. «Сохранить» — пользователь немедленно теряет доступ

---

### Загрузить документ для участников
1. `/admin/cms/documents` → «Добавить документ»
2. Выбрать файл (до 50 МБ)
3. Указать название и категорию
4. Поставить «Опубликовано» — документ появится на `/documents`

## 13. Техническая эксплуатация

### Адреса (разработка)

| Сервис | Адрес |
|---|---|
| Фронтенд | http://localhost:5173 |
| Бэкенд API | http://localhost:3002/api |
| PostgreSQL | localhost:5432 |

### Запуск в разработке

```bash
# Из корня monorepo
npm run dev
```
Запускает Docker (БД), NestJS бэкенд и Vite фронтенд одновременно.

### Конфигурация — `backend/.env`

```env
NODE_ENV=development
PORT=3002

# База данных
DB_HOST=localhost
DB_PORT=5432
DB_USER=sochigu
DB_PASS=sochigu_pass
DB_NAME=sochigu_contest
DB_SYNCHRONIZE=true        # true = TypeORM автоматически обновляет схему БД

# JWT токены
JWT_SECRET=change_me
JWT_EXPIRES_IN=15m
JWT_REFRESH_SECRET=change_me_refresh
JWT_REFRESH_EXPIRES_IN=7d

# Адрес фронтенда (для CORS и email-ссылок)
FRONTEND_URL=http://localhost:5173

# SMTP (оставьте закомментированным для отключения email)
# Без SMTP: регистрация работает без подтверждения email
# SMTP_HOST=smtp.yandex.ru
# SMTP_PORT=465
# SMTP_USER=your@email.ru
# SMTP_PASS=your_password

# Загрузка файлов
UPLOAD_DIR=./uploads
MAX_FILE_SIZE_MB=50
```

> `DB_SYNCHRONIZE=true` удобно в разработке, но **в production** лучше использовать миграции.

### Лимиты запросов (rate limiting)

| Эндпоинт | Лимит | Окно |
|---|---|---|
| POST /api/auth/register | 5 (prod) / 100 (dev) | 1 час |
| POST /api/contact-messages | 5 | 15 минут |
| Все остальные | 200 (prod) / 2000 (dev) | 15 минут |

### Хранение файлов

Все загруженные файлы сохраняются в `backend/uploads/` и **не входят в git**.

При деплое на новый сервер папку `uploads/` необходимо либо перенести, либо подключить как постоянный volume (в Docker).

```
uploads/
├── docs/          # Документы участников (до 50 МБ, .pdf .docx .xlsx и т.д.)
├── news/          # Обложки новостей (до 5 МБ, .jpg .png .webp .heic)
├── winners/       # Фото победителей (до 5 МБ)
├── experts/       # Фото экспертов (до 5 МБ)
└── applications/  # Файлы к заявкам (до 50 МБ, до 6 файлов на заявку)
```

### Резервное копирование БД

```bash
# Создать бэкап
docker exec sochigu_db pg_dump -U sochigu sochigu_contest > backup_$(date +%Y%m%d).sql

# Восстановить из бэкапа
docker exec -i sochigu_db psql -U sochigu sochigu_contest < backup_20260101.sql
```
